### 一、基础概念

#### 进程

一个单独运行的程序就是一个独立的进程，进程之间是相互独立存在的。比如打开的QQ、酷狗播放器、电脑管家等等都是独立的进程。

#### 线程

进程想要执行任务需要依赖线程，换句话说就是进程中的最小执行单元就是线程，并且一个进程中至少有一个线程。

#### 线程安全

某段代码在多线程下，**不管通过怎样的调用方式或者说这些线程如何交替的执行**，都能按照我们预期的行为去执行，就说这段代码是线程安全的。

下面的代码就是线程不安全的：

```
int i= 0; //全局变量
void increase() {
	i++;
}
```

上面increase函数就是线程不安全的。

#### 线程同步

同步就是协同步调，按预定的先后次序进行运行。如：你说完，我再说。这里的同步千万不要理解成那个同时进行，应是指协同、协助、互相配合。线程同步是指多线程通过特定的设置（如互斥量）来控制线程之间的执行顺序（即所谓的同步）也可以说是在线程之间通过同步建立起执行顺序的关系，如果没有同步，那线程之间是各自运行各自的！

线程互斥是指对于共享的进程系统资源，在各单个线程访问时的排它性。当有若干个线程都要使用某一共享资源时，任何时刻最多只允许一个线程去使用，其它要使用该资源的线程必须等待，直到占用资源者释放该资源。线程互斥可以看成是一种特殊的线程同步。

常见的线程同步技术就是`加锁`。

#### 临界区（Critical Section）

对共享资源访问的代码称为：`临界区`。一般需要被锁包围。

### 二、什么是锁

在计算机科学中，锁是一种同步机制，用于多线程环境中对资源访问的限制。

比如：

对于一个资源不能同时进行写操作。否则可能会造成数据混乱，丢失等。

对于一个资源不能同时一个线程在进行写操作，另一个线程在进行读操作。否则另一个线程可能读到的是旧的，甚至是已经被删除的数据。

这些情况都需要锁来进行对资源访问的限制。

### 三、iOS中的锁

在iOS中，有互斥锁、递归锁、条件变量、读写锁、自旋锁、信号量。

#### os_unfair_lock

OSSpinLock的替代品，iOS10.0可用。

特点

互斥锁，加锁失败时线程进入休眠状态等待唤醒。而不是忙等。

说明：

```
 * @discussion
 * Replacement for the deprecated OSSpinLock. Does not spin on contention but
 * waits in the kernel to be woken up by an unlock.
 *
 * As with OSSpinLock there is no attempt at fairness or lock ordering, e.g. an
 * unlocker can potentially immediately reacquire the lock before a woken up
 * waiter gets an opportunity to attempt to acquire the lock. This may be
 * advantageous for performance reasons, but also makes starvation of waiters a
 * possibility.
```

os_unfair_lock是OSSpinLock的替代品，但是与OSSpinLock不同的是，os_unfair_lock加锁失败时不会进入自旋状态而是进入休眠状态等待被唤醒。和OSSpinLock相同的是，os_unfair_lock也不会尝试公平加锁或按照顺序加锁，比如解锁线程可以在其他被唤醒的线程尝试加锁前又迅速的重新获得锁，这可能会带来一定的性能提升，但是也可能会造成等待线程一直得不到锁被饿死。

注意：

```
* This lock must be unlocked from the same thread that locked it, attempts to
* unlock from a different thread will cause an assertion aborting the process.
 
* Must be initialized with OS_UNFAIR_LOCK_INIT
```

**os_unfair_lock必须由获得锁的线程解锁（即加解锁必须是同一个线程），如果一个线程没有获得锁但却尝试解锁将导致崩溃**。

必须使用OS_UNFAIR_LOCK_INIT进行初始化。

主要API：

```
self.unfair_lock = OS_UNFAIR_LOCK_INIT; //初始化
void os_unfair_lock_lock(os_unfair_lock_t lock);
void os_unfair_lock_unlock(os_unfair_lock_t lock);
bool os_unfair_lock_trylock(os_unfair_lock_t lock);
```

使用示例：

```objc
@property (nonatomic, assign) os_unfair_lock unfair_lock; //assign

- (void)testos_unfair_lock {
    self.unfair_lock = OS_UNFAIR_LOCK_INIT;
    
    for (int k = 0; k < 5; k++) {
        dispatch_async(dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0), ^{
            NSLog(@"hello,线程：%@", [NSThread currentThread]);
            os_unfair_lock_lock(&self->_unfair_lock);
            [self increaseSomething];
            os_unfair_lock_unlock(&self->_unfair_lock);
            NSLog(@"bye,线程：%@", [NSThread currentThread]);
        });
    }
    dispatch_after(dispatch_time(DISPATCH_TIME_NOW, (int64_t)(5 * NSEC_PER_SEC)), dispatch_get_main_queue(), ^{
        NSLog(@"i = %ld", self.i);
    });
}

- (void)increaseSomething {
    NSLog(@"执行开始，线程：%@", [NSThread currentThread]);
    for (int k = 0; k < 1000; k++) {
        self.i++;
    }
    NSLog(@"执行结束，线程：%@", [NSThread currentThread]);
}
```

#### pthread_mutex_t（互斥锁）

mutex叫做”互斥锁”，等待锁的线程会处于休眠状态。需要导入头文件#import <pthread.h> 

特点：当一个线程获得锁后，其他线程将加锁失败从而进入休眠状态等待唤醒。这个特点既是优点也是缺点。

主要API：

```c
//初始化互斥锁
int pthread_mutex_init(pthread_mutex_t * __restrict,
		const pthread_mutexattr_t * _Nullable __restrict);
//加锁
int pthread_mutex_lock(pthread_mutex_t *);

//尝试加锁，加锁失败时不阻塞线程会返回一个错误码
int pthread_mutex_trylock(pthread_mutex_t *);

//解锁
int pthread_mutex_unlock(pthread_mutex_t *);

//销毁互斥锁
int pthread_mutex_destroy(pthread_mutex_t *);

//初始化互斥锁属性（初始化互斥锁时可以传入互斥锁属性）
int pthread_mutexattr_init(pthread_mutexattr_t *);

//互斥锁属性设置类型
int pthread_mutexattr_settype(pthread_mutexattr_t *, int);

/*
 * Mutex type attributes
 */
#define PTHREAD_MUTEX_NORMAL		0  //普通锁
#define PTHREAD_MUTEX_ERRORCHECK	1 //检错锁
#define PTHREAD_MUTEX_RECURSIVE		2  //递归锁
#define PTHREAD_MUTEX_DEFAULT		PTHREAD_MUTEX_NORMAL

//销毁互斥锁属性
int pthread_mutexattr_destroy(pthread_mutexattr_t *) __DARWIN_ALIAS(pthread_mutexattr_destroy);

/*
 * Mutex policy attributes
 */
#define PTHREAD_MUTEX_POLICY_FAIRSHARE_NP   1  //fair share 公平策略
#define PTHREAD_MUTEX_POLICY_FIRSTFIT_NP    3  //firt fit 最先适合策略

//互斥锁属性设置policy，默认是PTHREAD_MUTEX_POLICY_FIRSTFIT_NP
int pthread_mutexattr_setpolicy_np(pthread_mutexattr_t *, int);

//互斥锁属性设置prioceiling(优先级上限) 有效值-999~999，默认是0
int pthread_mutexattr_setprioceiling(pthread_mutexattr_t *, int);
```

使用步骤：

1. 初始化锁的属性

   ```
   pthread_mutexattr_t attr;
   pthread_mutexattr_init(&attr);
   pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_NORMAL);
   ```

2. 初始化锁

   ```
   pthread_mutex_init(&_mutex_lock, &attr);
   ```

3. 初始化锁结束以后，销毁属性

   ```
   pthread_mutexattr_destroy(&attr);
   ```

4. 加锁解锁

   ```
   pthread_mutex_lock(&self->_mutex_lock);
   [self increaseSomething];
   pthread_mutex_unlock(&self->_mutex_lock);
   ```

5. 销毁锁

   ```
   pthread_mutex_destroy(&self->_mutex_lock);
   ```

这里着重说明一下pthread_mutex互斥锁的三种类型：

PTHREAD_MUTEX_NORMAL：普通锁，当一个线程加锁以后，其余请求锁的线程将形成一个等待队列，并在解锁后按优先级获得锁。对于普通锁**同一线程**在未解锁时不能进行重复加锁，否则导致死锁。

PTHREAD_MUTEX_ERRORCHECK：检错锁，如果同一个线程请求同一个锁，则返回 EDEADLK （线程不会阻塞），否则与普通锁类型动作相同。这样就保证当不允许多次加锁时不会出现嵌套情况下的死锁。EDEADLK的定义：`#define EDEADLK     11       /* Resource deadlock avoided */`

PTHREAD_MUTEX_RECURSIVE：递归锁，允许**同一个线程**对**同一把锁**进行重复加锁。递归锁内部维护着一个Lock和一个counter变量，counter记录了acquire的次数，从而使得资源可以被多次require。直到一个线程所有的acquire都被release，其他的线程才能获得锁。使用递归锁时需要加解锁配对，如果解锁次数少了，锁将不会被释放，其他线程将一直处于等待状态而拿不到锁。

对于互斥锁属性

```c
int

	attr->prioceiling = _PTHREAD_DEFAULT_PRIOCEILING;
```

> 互斥锁解锁后哪个线程会被最先唤醒

The default scheduling policy, `SCHED_OTHER`, does not specify the order in which threads can acquire a lock. When multiple threads are waiting for a mutex, the order of acquisition is undefined. When there is contention, the default behavior is to unblock threads in priority order.

**pthread_mutex_unlock的函数说明：**

```
The pthread_mutex_unlock() function unlocks the mutex mutex. The mutex should be owned by the calling thread. If there are threads blocked on the mutex then the highest priority waiting thread is unblocked and becomes the next owner of the mutex.

If the thread's priority was raised when a higher-priority thread attempted to lock the mutex, the calling thread's priority is again adjusted when it unlocks the mutex; its new priority is the maximum of its own priority and the priorities of those threads it still blocks, either directly or indirectly. For more information, see “Mutexes: mutual exclusion locks” in the QNX Neutrino Microkernel chapter of the System Architecture guide.

If mutex has been locked more than once, it must be unlocked the same number of times before the next thread is given ownership of the mutex. This works only for recursive mutexes.

返回值：
EOK  //0
Success.
EINVAL //22
Invalid mutex mutex.
EPERM  //1
The current thread doesn't own the mutex.
```

综上，互斥锁解锁后，如果有多个线程处于等待状态，则优先级高的线程将获得锁。另外，调用该解锁函数的线程最好已经获得锁，一个线程如果没有获得互斥锁却调用解锁方法行为是未定义的，视锁的类型，有的会解锁成功并返回 0，有的会返回一个错误码EPERM，不会像 os_unfair_lock 那样直接崩溃。

示例：

多线程i++

对于普通互斥锁在未解锁时不能进行重复加锁，否则导致死锁：

```objc
- (void)testpthread_mutex_lock_normal_type_deadlock {
    pthread_mutexattr_t attr;
    pthread_mutexattr_init(&attr);
    pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_NORMAL);
    pthread_mutex_init(&_mutex_lock, &attr);
    pthread_mutexattr_destroy(&attr);
    
    for (int k = 0; k < 5; k++) {
        dispatch_async(dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0), ^{
            NSLog(@"hello,线程：%@", [NSThread currentThread]);
            pthread_mutex_lock(&self->_mutex_lock); //加锁
            NSLog(@"---1---,线程：%@", [NSThread currentThread]);
            pthread_mutex_lock(&self->_mutex_lock); //未解锁又加锁，导致线程自身休眠，无法执行后续解锁代码，从而造成死锁。
            NSLog(@"---2---,线程：%@", [NSThread currentThread]);
            [self increaseSomething];
            NSLog(@"---3---,线程：%@", [NSThread currentThread]);
            pthread_mutex_unlock(&self->_mutex_lock);
            NSLog(@"---4---,线程：%@", [NSThread currentThread]);
            pthread_mutex_unlock(&self->_mutex_lock);
            NSLog(@"bye,线程：%@", [NSThread currentThread]);
        });
    }
    dispatch_after(dispatch_time(DISPATCH_TIME_NOW, (int64_t)(5 * NSEC_PER_SEC)), dispatch_get_main_queue(), ^{
        NSLog(@"i = %ld", self.i);
    });
}
```

如果需要重复加锁，则可以使用PTHREAD_MUTEX_RECURSIVE递归锁。

将其改为递归锁，避免上述死锁问题：

```objc
    pthread_mutex_init(&_mutex_lock, &attr);

    for (int k = 0; k < 5; k++)
    {
```

再来看一个经典的死锁问题--科学家吃面问题：

有几个科学家在一张桌子旁，桌子上只有一双筷子和一碗面，我们将面和筷子都加锁。这时可能会出现一个科学家抢到面，另一个科学家抢到筷子的情况，这种情况下两个科学家都吃不了面，但双方都在等待对方吃面后释放资源，结果全部都阻塞，这就是死锁了。此时可以使用递归锁解决。

先看出现死锁的代码：

因为使用的是普通互斥锁，所以需要两把，一把用来给面加锁，一把用来给筷子加锁。

```objc

    pthread_mutex_init(&noodlesMutexLock, NULL);

        pthread_mutex_lock(&chopsticksMutexLock);
```

使用递归锁：

只需要定义一把递归锁，因为递归锁可以重复加锁。注意如果这里使用两把递归锁，反而会失去递归锁的效果，仍然会死锁。所以这里只需要定义一把递归锁。

```objc
- (void)test_pthread_recursive_lock_eat_noodles {
    __block pthread_mutex_t eatNoodlesMutexLock;
    pthread_mutexattr_t attr;
    pthread_mutexattr_init(&attr);
    pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE);
    pthread_mutex_init(&eatNoodlesMutexLock, &attr);
    
    dispatch_async(dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0), ^{
        NSLog(@"小红预备开始,线程：%@", [NSThread currentThread]);
        pthread_mutex_lock(&eatNoodlesMutexLock);
        NSLog(@"小红拿到筷子,线程：%@", [NSThread currentThread]);
        pthread_mutex_lock(&eatNoodlesMutexLock);
        NSLog(@"小红拿到面,线程：%@", [NSThread currentThread]);
        NSLog(@"小红吃面！！！！,线程：%@", [NSThread currentThread]);
        pthread_mutex_unlock(&eatNoodlesMutexLock);
        pthread_mutex_unlock(&eatNoodlesMutexLock);
    });
    
    dispatch_async(dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0), ^{
        NSLog(@"小明预备开始,线程：%@", [NSThread currentThread]);
        pthread_mutex_lock(&eatNoodlesMutexLock);
        NSLog(@"小明拿到面,线程：%@", [NSThread currentThread]);
        sleep(1);
        pthread_mutex_lock(&eatNoodlesMutexLock);
        NSLog(@"小明拿到筷子,线程：%@", [NSThread currentThread]);
        NSLog(@"小明吃面！！！！,线程：%@", [NSThread currentThread]);
        pthread_mutex_unlock(&eatNoodlesMutexLock);
        pthread_mutex_unlock(&eatNoodlesMutexLock);
    });
}
```

Q：使用普通互斥锁实现一个递归锁？

刚开始的时候，我是使用了一个普通互斥锁和一个条件变量实现的，后来看到别的实现发现使用一个普通互斥锁就已经够了。普通互斥锁替换成信号量也是可以的。

关键变量：

```
@property (nonatomic, assign) pthread_mutex_t mutexLock; //互斥锁
@property (nonatomic, assign) pthread_t curThread; //记录当前拥有锁的线程
@property (nonatomic, assign) int lockCount;  //记录加锁次数
```

实现：

```objc
- (instancetype)init {
    self = [super init];
    if (self) {
        pthread_mutex_init(&_mutexLock, NULL);
        _curThread = NULL;
        _lockCount = 0;
    }
    return self;
}

- (void)dealloc {
    pthread_mutex_destroy(&_mutexLock);
    _curThread = NULL;
}

- (void)lock {
    pthread_t thread = pthread_self();
    if (_curThread == thread) {
       _lockCount++;
    } else {
        pthread_mutex_lock(&_mutexLock);
      	//只有锁住后，才能去修改curThread和lockCount，否则线程不安全。
        _curThread = thread; 
        _lockCount = 1;
    }
}

- (int)tryLock {
    int ret = 0;
    pthread_t thread = pthread_self();
    if (_curThread == thread) {
       _lockCount++;
    } else {
        ret = 1;
    }
    
    return ret;
}

- (void)unlock {
    pthread_t thread = pthread_self();
    if (_curThread == thread) {
       _lockCount--;
        if (_lockCount == 0) {
            _curThread = NULL;
            pthread_mutex_unlock(&_mutexLock);
        }
    }
}
```

#### pthread_cond_t（条件变量）

前面提到互斥锁的特点：一个线程获得锁后，其他线程只能休眠等待。而条件变量则允许线程阻塞并等待另一个线程发送唤醒信号从而弥补互斥锁的不足，它常和互斥锁一起使用。

主要API：

```c
//初始化条件,条件属性参数可以传NULL
int pthread_cond_init(
		pthread_cond_t * __restrict,
		const pthread_condattr_t * _Nullable __restrict)
		__DARWIN_ALIAS(pthread_cond_init);

//激活一个等待该条件的线程
int pthread_cond_signal(pthread_cond_t *);

//激活所有等待该条件的线程
int pthread_cond_broadcast(pthread_cond_t *);

//等待条件
int pthread_cond_wait(pthread_cond_t * __restrict,
		pthread_mutex_t * __restrict) __DARWIN_ALIAS_C(pthread_cond_wait);

//等待条件,可以指定等待的时长
int pthread_cond_timedwait(
		pthread_cond_t * __restrict, pthread_mutex_t * __restrict,
		const struct timespec * _Nullable __restrict)
		__DARWIN_ALIAS_C(pthread_cond_timedwait);

//销毁条件
int pthread_cond_destroy(pthread_cond_t *);
```

重要API说明：

**pthread_cond_wait**

```
int pthread_cond_wait(pthread_cond_t *cv, pthread_mutex_t *mutex);
```

1.把当前线程放到等待条件的线程列表上（即通过条件变量传递消息的线程列表，可能多个线程同时等待）。

2.解锁mutex参数指向的互斥锁，并使当前线程阻塞（线程进入休眠状态）在cv参数指向的条件变量上。

3.pthread_cond_wait函数在即将返回前会重新加锁，当pthread_cond_wait函数**真正返回**后由于已经重新加锁这也相当于互斥锁被当前线程重新获得。

**pthread_cond_wait函数的返回并不意味着条件的值一定发生了变化**，必须重新检查条件的值。最好的测试方法是循环调用pthread_cond_wait函数，并把满足条件的表达式置为循环的终止条件。

被阻塞的线程可以被pthread_cond_signal函数，pthread_cond_broadcast函数唤醒，也可能在被信号中断后被唤醒。注意：发出pthread_cond_signal的线程如果没有解锁，那么被唤醒的线程依然处于阻塞状态直到拥有锁的线程解锁。（过程是这样子的：pthread_cond_wait被pthread_cond_signal唤醒，于是重新加锁，但此时锁还没有被解锁于是又休眠了）

阻塞在同一个条件变量上的不同线程被唤醒的次序是不一定的，取决于线程调度策略。

pthread_cond_wait必须要在mutex的锁定区域内使用。

**要想知道为什么是这样子还是那句话看源码！！！**

**pthread_cond_signal**

```
int pthread_cond_signal(pthread_cond_t *cv);
```

pthread_cond_signal函数的作用是解除**一个**等待参数cv所指定的条件变量的线程的阻塞状态。
唤醒阻塞在条件变量上的所有线程的顺序由调度策略决定，如果线程的调度策略是SCHED_OTHER类型的，系统将根据线程的优先级唤醒线程。
如果没有线程被阻塞在条件变量上，那么调用pthread_cond_signal()将没有作用。

pthread_cond_signal函数是否必须在临界区里调用，仍有争议。分场景，如果不想因为那么一点点的性能提升而思前想后，那么放在临界区里调用准没错。

**pthread_cond_broadcast**

对所有等待参数cond所指定的条件变量的线程解除阻塞，调用成功返回0，否则返回错误代码。

#### pthread_rwlock_t（读写锁）

前面mutex锁有个缺点，就是只要锁住了，不管其他线程要干什么，都不允许进入临界区。设想这样一种情况：临界区foo变量在被bar1线程读着，加了个mutex锁，bar2线程如果也要读foo变量，因为被bar1加了个互斥锁，那就不能读了。但事实情况是，读取数据不影响数据内容本身，所以即便被1个线程读着，另外一个线程也应该允许他去读。除非另外一个线程是写操作，为了避免数据不一致的问题，写线程就需要等读线程都结束了再写。

读写锁也叫做**共享-独占锁**，当读写锁以读模式锁住时，它是以共享模式锁住的，当它以写模式锁住时，它是以独占模式锁住的。

读写锁的加锁规则：

1 如果没有加写锁时，那么多个线程可以同时加读锁；如果有加写锁时，不可以加读、写锁

2 不管是加了读锁还是写锁，都不能继续加写锁。

**pthread_rwlock_rdlock**

如果当前**没有写线程持有锁**或者**没有写线程阻塞在取锁**时则可以立刻获取到锁，否则请求锁的线程阻塞等待。

注意：

如果在进行调用时，调用线程持有 **rwlock** 中的写锁，则结果是不确定的。

为避免写入器资源匮乏，一般使写入器的优先级高于读取器。

**pthread_rwlock_wrlock**

如果当前没有其他读线程或写线程持有读写锁 **rwlock**，则调用线程将获取写锁。否则，调用线程将阻塞。

注意：

如果在进行调用时，调用线程持有读/写锁，则结果是不确定的。结果不确定指的是不同语言可能因实现不同而外在表现也不同。有的可能就死锁了，而有的可能会锁降级。

**锁降级**

锁降级指的是一个线程先拥有写锁，然后拥有读锁，在对数据处理完成后，释放写锁，此时该线程所持有的锁是读锁，从写锁降级为读锁。

疑问：

Q1：为什么多个读线程可以重复获得同一个读锁？如果其中某个读线程读加锁后没有解锁会怎样？

如果其中某个读线程读加锁后没有解锁，那么写线程将一直等待直到所有读锁解锁。读加锁和解加锁的次数一定要相等。

Q2：一个已经拥有读锁的读线程能不能重入读锁？

如果当前没有等待的写线程，则读线程自身可以重入读锁，否则将导致死锁。这个其实很危险，因为一旦有等待的写线程就会造成死锁。所以最好是先解锁再加锁。再次重申，读加锁和解加锁的次数一定要相等。

Q3：一个已经拥有写锁的写线程能不能重入写锁？

重入时会加锁失败并立即返回EDEADLK，不会导致线程自身死锁。本次加写锁相当于无效操作。后面解锁时只能解锁一次，因为重入写锁的那一次是无效加锁。否则会有问题。失败的加锁操作不能对应一次解锁操作。

```
#define EDEADLK         11              /* Resource deadlock avoided */
                                        /* 11 was EAGAIN */
```

Q4：一个已经拥有读锁的读线程能不能继续加写锁？

不能，会导致死锁。因为加写锁必须要所有锁都释放。

Q5：一个已经拥有写锁的写线程能不能继续加读锁？

继续加读锁会失败并立即返回EDEADLK，不会导致线程自身死锁。可以理解为能够继续加读锁。后面解锁时也只能解锁一次，因为有一次加锁是失败的。否则会有问题。因为写锁是独占的所以该线程继续加读锁或写锁都没有啥问题，而拥有读锁的线程则限制很多。

虽然读写锁在一定条件下允许重入，但还是不推荐这种嵌套调用，很容易出问题。

Q6：使用互斥锁和条件变量实现一个读写锁？或者其他什么锁等实现一个读写锁？

总之，读的时候想办法把要写的线程整休眠了（时机就是加写锁的时候），写的时候想办法把要读的线程整休眠（时机就是加读锁的时候），然后在操作完后（时机就是unlock的时候）根据读优先还是写优先唤醒对应线程。

#### dispatch_barrier_async

栅栏技术

对于上面的读多写少的问题，GCD提供了一种简便的方法dispatch_barrier_async函数。这个函数传入的队列必须是自己通过dispatch_queue_cretate创建的并发队列，如果传入的是一个串行或是一个全局的并发队列，那这个函数便等同于dispatch_async函数的效果。

dispatch_barrier_asynch函数的处理流程：

```
 读任务    | 由dispatch_barrier_async追加的处理   |   读任务
  读任务   |          写任务                     |     读任务
   读任务  |        同时只执行一个                 |      读任务
```

dispatch_barrier_async函数会等待追加到并发队列上的并行执行的处理全部结束之后，再将指定的处理追加到该并发队列中。然后再由dispatch_barrier_async追加的处理执行完毕之后，并发队列才恢复为一般的动作，后面的处理由开始并行执行。

使用dispatch_barrier_async可以实现高效率的数据库访问和文件访问。

#### NSLock

NSLock是对 pthread_mutex_t 普通互斥锁的一层封装，`pthread_mutex_init(mutex, nil)`。

NSLock 遵循 NSLocking 协议。lock 方法是加锁，unlock 是解锁，tryLock 是尝试加锁，如果失败的话返回 NO，lockBeforeDate: 是在指定Date之前尝试加锁，如果在指定时间之前都不能加锁，则返回NO

源码：

```swift
open class NSLock: NSObject, NSLocking {
    internal var mutex = _MutexPointer.allocate(capacity: 1)
#if os(macOS) || os(iOS) || os(Windows)
    private var timeoutCond = _ConditionVariablePointer.allocate(capacity: 1)
    private var timeoutMutex = _MutexPointer.allocate(capacity: 1)
#endif

    public override init() {
#if os(Windows)
        InitializeSRWLock(mutex)
        InitializeConditionVariable(timeoutCond)
        InitializeSRWLock(timeoutMutex)
#else
        pthread_mutex_init(mutex, nil)
#if os(macOS) || os(iOS)
        pthread_cond_init(timeoutCond, nil)
        pthread_mutex_init(timeoutMutex, nil)
#endif
#endif
    }
    
    deinit {
#if os(Windows)
        // SRWLocks do not need to be explicitly destroyed
#else
        pthread_mutex_destroy(mutex)
#endif
        mutex.deinitialize(count: 1)
        mutex.deallocate()
#if os(macOS) || os(iOS) || os(Windows)
        deallocateTimedLockData(cond: timeoutCond, mutex: timeoutMutex)
#endif
    }
    
    open func lock() {
#if os(Windows)
        AcquireSRWLockExclusive(mutex)
#else
        pthread_mutex_lock(mutex)
#endif
    }

    open func unlock() {
#if os(Windows)
        ReleaseSRWLockExclusive(mutex)
        AcquireSRWLockExclusive(timeoutMutex)
        WakeAllConditionVariable(timeoutCond)
        ReleaseSRWLockExclusive(timeoutMutex)
#else
        pthread_mutex_unlock(mutex)
#if os(macOS) || os(iOS)
        // Wakeup any threads waiting in lock(before:)
        pthread_mutex_lock(timeoutMutex)
        pthread_cond_broadcast(timeoutCond)
        pthread_mutex_unlock(timeoutMutex)
#endif
#endif
    }

    open func `try`() -> Bool {
#if os(Windows)
        return TryAcquireSRWLockExclusive(mutex) != 0
#else
        return pthread_mutex_trylock(mutex) == 0
#endif
    }
    
    open func lock(before limit: Date) -> Bool {
#if os(Windows)
        if TryAcquireSRWLockExclusive(mutex) != 0 {
          return true
        }
#else
        if pthread_mutex_trylock(mutex) == 0 {
            return true
        }
#endif

#if os(macOS) || os(iOS) || os(Windows)
        return timedLock(mutex: mutex, endTime: limit, using: timeoutCond, with: timeoutMutex)
#else
        guard var endTime = timeSpecFrom(date: limit) else {
            return false
        }
        return pthread_mutex_timedlock(mutex, &endTime) == 0
#endif
    }

    open var name: String?
}

extension NSLock {
    internal func synchronized<T>(_ closure: () -> T) -> T {
        self.lock()
        defer { self.unlock() }
        return closure()
    }
}
```

#### NSCondition

NSCondition是对普通互斥锁和条件变量的封装。如果对 pthread 里的几种锁很熟悉的话这里的就很简单了。

源码：

```swift
open class NSCondition: NSObject, NSLocking {
    internal var mutex = _MutexPointer.allocate(capacity: 1)
    internal var cond = _ConditionVariablePointer.allocate(capacity: 1)

    public override init() {
#if os(Windows)
        InitializeSRWLock(mutex)
        InitializeConditionVariable(cond)
#else
        pthread_mutex_init(mutex, nil)
        pthread_cond_init(cond, nil)
#endif
    }
    
    deinit {
#if os(Windows)
        // SRWLock do not need to be explicitly destroyed
#else
        pthread_mutex_destroy(mutex)
        pthread_cond_destroy(cond)
#endif
        mutex.deinitialize(count: 1)
        cond.deinitialize(count: 1)
        mutex.deallocate()
        cond.deallocate()
    }
    
    open func lock() {
#if os(Windows)
        AcquireSRWLockExclusive(mutex)
#else
        pthread_mutex_lock(mutex)
#endif
    }
    
    open func unlock() {
#if os(Windows)
        ReleaseSRWLockExclusive(mutex)
#else
        pthread_mutex_unlock(mutex)
#endif
    }
    
    open func wait() {
#if os(Windows)
        SleepConditionVariableSRW(cond, mutex, WinSDK.INFINITE, 0)
#else
        pthread_cond_wait(cond, mutex)
#endif
    }

    open func wait(until limit: Date) -> Bool {
#if os(Windows)
        return SleepConditionVariableSRW(cond, mutex, timeoutFrom(date: limit), 0)
#else
        guard var timeout = timeSpecFrom(date: limit) else {
            return false
        }
        return pthread_cond_timedwait(cond, mutex, &timeout) == 0
#endif
    }
    
    open func signal() {
#if os(Windows)
        WakeConditionVariable(cond)
#else
        pthread_cond_signal(cond)
#endif
    }
    
    open func broadcast() {
#if os(Windows)
        WakeAllConditionVariable(cond)
#else
        pthread_cond_broadcast(cond)
#endif
    }
    
    open var name: String?
}
```

#### NSConditionLock

NSConditionLock 是对 NSCondition 的进一步封装。内部有三个变量`_cond`，`_value`，和`_thread`。

源码实现：

```swift
open class NSConditionLock : NSObject, NSLocking {
    internal var _cond = NSCondition()
    internal var _value: Int
    internal var _thread: _swift_CFThreadRef?
    
    public convenience override init() {
        self.init(condition: 0)
    }
    
    public init(condition: Int) {
        _value = condition
    }

    open func lock() {
        let _ = lock(before: Date.distantFuture)
    }

    open func unlock() {
        _cond.lock()
#if os(Windows)
        _thread = INVALID_HANDLE_VALUE
#else
        _thread = nil
#endif
        _cond.broadcast()
        _cond.unlock()
    }
    
    open var condition: Int {
        return _value
    }

    open func lock(whenCondition condition: Int) {
        let _ = lock(whenCondition: condition, before: Date.distantFuture)
    }

    open func `try`() -> Bool {
        return lock(before: Date.distantPast)
    }
    
    open func tryLock(whenCondition condition: Int) -> Bool {
        return lock(whenCondition: condition, before: Date.distantPast)
    }

    open func unlock(withCondition condition: Int) {
        _cond.lock()
#if os(Windows)
        _thread = INVALID_HANDLE_VALUE
#else
        _thread = nil
#endif
        _value = condition
        _cond.broadcast()
        _cond.unlock()
    }

    open func lock(before limit: Date) -> Bool {
        _cond.lock()
        while _thread != nil {
            if !_cond.wait(until: limit) {
                _cond.unlock()
                return false
            }
        }
#if os(Windows)
        _thread = GetCurrentThread()
#else
        _thread = pthread_self()
#endif
        _cond.unlock()
        return true
    }
    
    open func lock(whenCondition condition: Int, before limit: Date) -> Bool {
        _cond.lock()
        while _thread != nil || _value != condition {
            if !_cond.wait(until: limit) {
                _cond.unlock()
                return false
            }
        }
#if os(Windows)
        _thread = GetCurrentThread()
#else
        _thread = pthread_self()
#endif
        _cond.unlock()
        return true
    }
    
    open var name: String?
}
```

_cond：NSCondition类型

_value：整型类型，用于保存条件值。

_thread：记录当前加锁的线程。

`open func lock(before limit: Date) -> Bool`的实现：

1. 将_cond 加锁
2. 循环判断`_thread`是否有值，没有值则将线程自身赋值给`_thread`，然后将_cond 解锁并返回true 表示加锁成功。
3. 如果`_thread`有值表明当前已被其他线程上锁，于是调用` _cond.wait(until: limit)`进入休眠状态，该函数如果是超时则返回 NO，如果是被signal 唤醒则返回 YES。如果是超时则解锁并返回 false 表明加锁失败，如果是被signal 唤醒则重复 2。

`open func unlock(withCondition condition: Int)`的实现：

1. 将_cond 加锁
2. 将_thread置为 nil
3. 发出广播broadcast，唤醒所有等待的线程
4. 将_cond 解锁

这两个方法只是协议里定义的，此时 NSConditionLock 仅仅是当做一把普通的锁，效果等价于 NSLock。

真正发挥威力的是结合条件值来使用的时候。

`open func lock(whenCondition condition: Int)`的实现：

最终调用`open func lock(whenCondition condition: Int, before limit: Date) -> Bool`,它的实现：

1. 将_cond 加锁
2. 循环判断`_thread`的值是否等于 nil，`_value`的值是否等于传入的条件值
3. 如果`_thread`的值等于 nil，`_value`的值等于传入的条件值，则将线程自身赋值给`_thread`，然后将_cond 解锁并返回true 表示加锁成功。
4. 如果不符合条件则调用` _cond.wait(until: limit)`进入休眠等待状态

简而言之，就是如果条件相等且当前没有加锁的线程则加锁成功，否则休眠等待唤醒。

再看下`open func unlock(withCondition condition: Int)`的实现：

1. 将_cond 加锁
2. 将_thread置为 nil
3. 更改_value的值为传入的条件值
4. 发出广播broadcast，唤醒所有等待的线程
5. 将_cond 解锁

注意这里传入的condition的值只是用来更新内部的`_value` 的，并不是用于判断`_value`的值等于condition才解锁。

示例：

```objc
- (void)test_NSConditionLock1 {
    self.ns_condition_lock = [[NSConditionLock alloc] initWithCondition:0];
    
    int kHASDATA = 1;
    int kDONTHASDATA = 0;

    for (int i = 0; i < 10; i++) {
        dispatch_async(dispatch_get_global_queue(0, 0), ^{
//            NSLog(@"remove 线程 will lock");
            [self.ns_condition_lock lockWhenCondition:kHASDATA]; //消费者加锁，有数据时加锁成功开始消费
//            NSLog(@"remove 线程 did lock and will remove");
            sleep(1);
            id last = mArr.lastObject;
            [mArr removeObject:last];
            NSLog(@"obj被移除:%@", last);
//            NSLog(@"remove 线程 did remove and will unlock");
            int condition = mArr.count > 0 ? kHASDATA : kDONTHASDATA;
            [self.ns_condition_lock unlockWithCondition:condition]; //消费完毕，更新条件并发出信号并释放锁
        });
    }
    
    for (int i = 0; i < 10; i++) {
        dispatch_async(dispatch_get_global_queue(0, 0), ^{
//            NSLog(@"add 线程 will lock");
            [self.ns_condition_lock lockWhenCondition:kDONTHASDATA]; //生产者加锁，没有数据时加锁成功开始生产

            [mArr addObject:obj];

            [self.ns_condition_lock unlockWithCondition:kHASDATA]; //生产完毕，更新条件并发出信号并释放锁
        });
    }
}
```

#### 自旋锁的实现原理

自旋锁的目的是为了**确保临界区只有一个线程可以访问**，它的使用可以用下面这段伪代码来描述:

```
bool lock = 0; //0代表空闲状态，1代表锁住状态
{
		while (lock); //查询锁的状态，相当于申请锁
		lock = 1; //加锁
		临界区代码
		lock = 0; //解锁
}
```

刚开始时，锁处于空闲状态，线程A申请锁成功，然后加锁。当线程B也申请锁时，锁已经处于锁住状态，因此申请失败，于是线程B开始循环查询锁的状态，当线程A解锁后，线程B检测到锁处于空闲状态，于是申请成功，随即加上锁。

看似没什么问题，但是如果一开始就有两个线程申请锁，那么必然都申请成功，同时执行临界区代码，这样就没有做到对资源的访问限制了。

需要改进：

```c
bool lock = 0;
{
		while (testAndSet(&lock)); //尝试加锁
		临界区代码
		lock = 0; //解锁
}

bool testAndSet(bool *target) {
		bool old_value = *target;
		*target = true;
		return old_value;
}
```

这里testAndSet函数是test_and_set汇编指令的伪代码，test_and_set汇编指令是硬件层面保证的原子操作。这段代码的作用是把 target 的值设置为 1，并返回原来的值。如果当前锁为锁住状态，`*target = true;`无影响，尝试加锁失败依旧得忙等着。如果当前锁为空闲状态，则设置为锁住状态，并返回原来的值--空闲状态，于是加锁成功。由于是原子操作，所以不会同时有两个线程执行该函数。

可以看到，自旋锁会让加锁失败的线程进入忙等状态，白白浪费CPU资源。自旋锁的这一特点可以说即是优点也是缺点，衡量的标准就是看忙等耗时的长短，这取决于临界区的场景。如果临界区的代码很简单，那么忙等的时间就会很短，这时自旋锁的效率就很高。如果临界区代码是一些I/O操作，那么忙等的时间就会很长直到被系统抢占，这个时候自旋锁就大大的浪费了CPU资源。

如果低优先级线程先拿到锁，那么高优先级线程就会进入忙等(busy-wait)状态，消耗大量 CPU 时间，如果线程的调度算法是让高优先级线程始终在低优先级线程前执行，那么低优先级线程将拿不到 CPU 时间，也就无法完成任务并释放锁。这种问题被称为优先级反转。这也是iOS中OSSpinLock 不再安全的原因。在 iOS 10/macOS 10.12 发布时，苹果提供了新的 os_unfair_lock 作为 OSSpinLock 的替代，并且将 OSSpinLock 标记为了 Deprecated。

#### 死锁

所谓死锁： 是指两个或两个以上的进程或线程在执行过程中，因争夺资源而造成的一种互相等待的现象，若无外力作用，它们都将无法推进下去。此时称系统处于死锁状态或系统产生了死锁，这些永远在互相等待的进程称为死锁进程，

**产生死锁的原因**

死锁的四个必要条件

互斥条件：一个资源每次只能被一个进程使用。即在一段时间内某资源仅为一个进程所占有。此时若有其他进程请求该资源，则请求进程只能等待。

请求与保持条件：一个进程因请求资源而阻塞时，对已获得的资源保持不放。即进程已经保持了至少一个资源，但又提出了新的资源请求，而该资源已被其他进程占有，此时请求进程被阻塞，但对自己已获得的资源保持不放。

不可剥夺条件：进程所获得的资源在未使用完毕之前，不能被其他进程强行夺走。即只能由获得该资源的进程自己来释放（只能是主动释放)。

循环等待条件： 若干进程间形成首尾相接循环等待资源的关系。

这四个条件是死锁的必要条件，只要系统发生死锁，这些条件必然成立，而只要上述条件之一不满足，就不会发生死锁。

进程死锁和线程死锁的区别就是主语不同而已。

#### 死锁的预防与解决

在有些情况下死锁是可以避免的。三种用于避免死锁的技术：

1. 加锁顺序（线程按照一定的顺序加锁）
2. 加锁时限（线程尝试获取锁的时候加上一定的时限，超过时限则放弃对该锁的请求，并释放自己占有的锁）
3. 死锁检测
