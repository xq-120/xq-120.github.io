### 基础概念

#### 进程

一个单独运行的程序就是一个独立的进程，进程之间是相互独立存在的。比如打开的QQ、酷狗播放器、电脑管家等等都是独立的进程。

#### 线程

进程想要执行任务需要依赖线程，换句话说就是进程中的最小执行单元就是线程，并且一个进程中至少有一个线程。

#### 线程安全

某段代码在多线程下，**不管通过怎样的调用方式或者说这些线程如何交替的执行**，都能按照我们预期的行为去执行，就说这段代码是线程安全的。

下面的代码就是线程不安全的：

```
int i= 0; //全局变量
void increase() {
	i++;
}
```

上面increase函数就是线程不安全的。

#### 线程同步

同步就是协同步调，按预定的先后次序进行运行。如：你说完，我再说。这里的同步千万不要理解成那个同时进行，应是指协同、协助、互相配合。线程同步是指多线程通过特定的设置（如互斥量）来控制线程之间的执行顺序（即所谓的同步）也可以说是在线程之间通过同步建立起执行顺序的关系，如果没有同步，那线程之间是各自运行各自的！

线程互斥是指对于共享的进程系统资源，在各单个线程访问时的排它性。当有若干个线程都要使用某一共享资源时，任何时刻最多只允许一个线程去使用，其它要使用该资源的线程必须等待，直到占用资源者释放该资源。线程互斥可以看成是一种特殊的线程同步。

常见的线程同步技术就是`加锁`。

#### 临界区（Critical Section）

对共享资源访问的代码称为：`临界区`。一般需要被锁包围。

### 一、什么是锁

在计算机科学中，锁是一种同步机制，用于多线程环境中对资源访问的限制。

比如：

对于一个资源不能同时进行写操作。否则可能会造成数据混乱，丢失等。

对于一个资源不能同时一个线程在进行写操作，另一个线程在进行读操作。否则另一个线程可能读到的是旧的，甚至是已经被删除的数据。

这些情况都需要锁来进行对资源访问的限制。

### 二、iOS中的锁

在iOS中，有互斥锁、递归锁、读写锁、自旋锁、条件锁、信号量。

> 对同一个锁对象的加解锁必须保持在同一线程执行，如果尝试在不同线程去加解锁将会引发一个运行时的错误。

#### 读写锁

前面mutex锁有个缺点，就是只要锁住了，不管其他线程要干什么，都不允许进入临界区。设想这样一种情况：临界区foo变量在被bar1线程读着，加了个mutex锁，bar2线程如果也要读foo变量，因为被bar1加了个互斥锁，那就不能读了。但事实情况是，读取数据不影响数据内容本身，所以即便被1个线程读着，另外一个线程也应该允许他去读。除非另外一个线程是写操作，为了避免数据不一致的问题，写线程就需要等读线程都结束了再写。

#### 自旋锁的实现原理

自旋锁的目的是为了**确保临界区只有一个线程可以访问**，它的使用可以用下面这段伪代码来描述:

```
bool lock = 0; //0代表空闲状态，1代表锁住状态
{
		while (lock); //查询锁的状态，相当于申请锁
		lock = 1; //加锁
		临界区代码
		lock = 0; //解锁
}
```

刚开始时，锁处于空闲状态，线程A申请锁成功，然后加锁。当线程B也申请锁时，锁已经处于锁住状态，因此申请失败，于是线程B开始循环查询锁的状态，当线程A解锁后，线程B检测到锁处于空闲状态，于是申请成功，随即加上锁。

看似没什么问题，但是如果一开始就有两个线程申请锁，那么必然都申请成功，同时执行临界区代码，这样就没有做到对资源的访问限制了。

需要改进：

```c
bool lock = 0;
{
		while (testAndSet(&lock)); //尝试加锁
		临界区代码
		lock = 0; //解锁
}

bool testAndSet(bool *target) {
		bool status = *target;
		*target = true;
		return status;
}
```

这里testAndSet函数是test_and_set汇编指令的伪代码，test_and_set汇编指令是硬件层面保证的原子操作。这段代码的作用是把 target 的值设置为 1，并返回原来的值。如果当前锁为锁住状态，`*target = true;`无影响，尝试加锁失败依旧得忙等着。如果当前锁为空闲状态，则设置为锁住状态，并返回原来的值--空闲状态，于是加锁成功。由于是原子操作，所以不会同时有两个线程执行该函数。

可以看到，自旋锁会让加锁失败的线程进入忙等状态，白白浪费CPU资源。自旋锁的这一特点可以说即是优点也是缺点，衡量的标准就是看忙等耗时的长短，这取决于临界区的场景。如果临界区的代码很简单，那么忙等的时间就会很短，这时自旋锁的效率就很高。如果临界区代码是一些I/O操作，那么忙等的时间就会很长直到被系统抢占，这个时候自旋锁就大大的浪费了CPU资源。

如果低优先级线程先拿到锁，那么高优先级线程就会进入忙等(busy-wait)状态，消耗大量 CPU 时间，如果线程的调度算法是让高优先级线程始终在低优先级线程前执行，那么低优先级线程将拿不到 CPU 时间，也就无法完成任务并释放锁。这种问题被称为优先级反转。这也是iOS中OSSpinLock 不再安全的原因。在 iOS 10/macOS 10.12 发布时，苹果提供了新的 os_unfair_lock 作为 OSSpinLock 的替代，并且将 OSSpinLock 标记为了 Deprecated。

#### 产生死锁的原因

死锁的四个必要条件

互斥条件：一个资源每次只能被一个进程使用。即在一段时间内某资源仅为一个进程所占有。此时若有其他进程请求该资源，则请求进程只能等待。

请求与保持条件：一个进程因请求资源而阻塞时，对已获得的资源保持不放。即进程已经保持了至少一个资源，但又提出了新的资源请求，而该资源已被其他进程占有，此时请求进程被阻塞，但对自己已获得的资源保持不放。

不可剥夺条件：进程所获得的资源在未使用完毕之前，不能被其他进程强行夺走。即只能由获得该资源的进程自己来释放（只能是主动释放)。

循环等待条件： 若干进程间形成首尾相接循环等待资源的关系。

这四个条件是死锁的必要条件，只要系统发生死锁，这些条件必然成立，而只要上述条件之一不满足，就不会发生死锁。

进程死锁和线程死锁的区别就是主语不同而已。

#### 死锁的预防与解决

在有些情况下死锁是可以避免的。三种用于避免死锁的技术：

1. 加锁顺序（线程按照一定的顺序加锁）
2. 加锁时限（线程尝试获取锁的时候加上一定的时限，超过时限则放弃对该锁的请求，并释放自己占有的锁）
3. 死锁检测
